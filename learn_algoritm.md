# База по алгоритмам: что нужно знать, чтобы начать решать


---

## Слой 1. Что значит «алгоритм работает»

Прежде чем писать код, надо уметь отвечать на два вопроса про любой алгоритм:

**1. Он правильный?** То есть на всех допустимых входах даёт верный ответ. Не «на моём примере прошло» — а на всех. Сюда входят: пустой вход, один элемент, дубликаты, уже отсортированный, отсортированный наоборот, отрицательные числа, переполнение.

**2. Он достаточно быстрый?** Это измеряется в Big-O — количестве операций как функция от размера входа `n`. Если `n = 10⁵`, то `O(n²) = 10¹⁰` операций — это примерно 100 секунд, TLE. А `O(n log n) ≈ 1.7·10⁶` — меньше секунды.

Грубая шпаргалка «что пройдёт по времени за ~1 сек»:

```
n ≤ 10        → O(n!)      можно перебор всех перестановок
n ≤ 20        → O(2ⁿ)      битовые маски, подмножества
n ≤ 500       → O(n³)      трёхмерное DP, Флойд
n ≤ 5000      → O(n²)      двойной цикл
n ≤ 10⁵       → O(n log n) сортировка, отрезки, куча
n ≤ 10⁶       → O(n)       один проход
n ≤ 10⁹       → O(log n)   бинарный поиск, битовые операции
```

Это первое, что смотришь в условии: размер входа подсказывает допустимую сложность, а та — семейство подходящих алгоритмов.

## Слой 2. Структуры данных и что у них быстро

Без этого дальше бесполезно. Для каждой структуры нужно знать: что она хранит, и сколько стоят базовые операции.

```
Структура         доступ   вставка  удаление  поиск    заметка
──────────────────────────────────────────────────────────────────────
Массив            O(1)     O(n)     O(n)      O(n)     смежная память
Хеш-таблица       —        O(1)*    O(1)*     O(1)*    *в среднем
Связный список    O(n)     O(1)     O(1)      O(n)     при наличии указателя
Стек / Очередь    —        O(1)     O(1)      —        LIFO / FIFO
Куча (heap)       O(1)     O(log n) O(log n)  —        доступ только к min/max
BST сбалансир.    O(log n) O(log n) O(log n)  O(log n) TreeMap / std::set
Trie              —        O(L)     O(L)      O(L)     L = длина ключа
```

Главное — не сами операции, а **чувство, когда что применять**. Это приходит через паттерны (слой 4). Ключевые связки:

- `set / dict` — убрать `O(n)` поиск. Почти любая задача «найти/посчитать/дедуплицировать» начинается отсюда.
- `deque` — когда нужна очередь или стек с доступом с обеих сторон (BFS, sliding window maximum).
- `heap` — когда важен только минимум/максимум в динамической коллекции (top-K, Дейкстра, merge-K).
- `stack` — когда порядок обработки зависит от «последнего добавленного» (парсинг, monotonic stack, DFS итеративно).

## Слой 3. Две техники, которые лежат под всем остальным

### Рекурсия

Алгоритм, который вызывает сам себя на меньшей задаче. Каждая рекурсия — это:

```
base case       — когда ответ очевиден и мы просто его возвращаем
recursive case  — как свести текущую задачу к одной-двум меньшим
```

Пример — факториал:

```python
def fact(n):
    if n <= 1:      # база
        return 1
    return n * fact(n - 1)   # рекурсивный шаг
```

Ментальная модель: не пытайся «распутать» рекурсию в голове до самого дна. Доверься, что `fact(n-1)` вернёт правильный факториал, и строй `fact(n)` из этого. Это называется **принципом доверия рекурсии**, и без него backtracking, деревья и DP не пишутся.

### Инвариант цикла

Это утверждение, которое **истинно на каждой итерации цикла**. Доказать корректность цикла = доказать инвариант.

Пример — бинарный поиск:

> Инвариант: если `target` в массиве, то он лежит в `arr[l..r]`.

Проверка: изначально `l=0, r=n-1` — инвариант выполнен. На каждом шаге мы либо сужаем диапазон в сторону, где таргет точно не может быть, либо находим его. Значит, после выхода из цикла либо мы его нашли, либо его нет вообще.

Эта штука звучит академично, но реально спасает от off-by-one ошибок. Когда в бинарном поиске ломаешь голову «`l < r` или `l <= r`, `mid-1` или `mid`» — ответом всегда является «какой инвариант я поддерживаю».

## Слой 4. Паттерны решения

Это уже прикладной уровень — то, что показано в [моей предыдущей визуализации](#). Опытный решающий при чтении условия **сразу узнаёт паттерн по ключевым словам**:

| Слышишь в условии | Думаешь |
|---|---|
| «отсортированный массив» + «пара/тройка» | Two Pointers или Binary Search |
| «подстрока/подмассив с условием» | Sliding Window |
| «все комбинации / перестановки / пути» | Backtracking |
| «k-й наибольший / top-K / поток данных» | Heap |
| «кратчайший путь / связность / компоненты» | Graphs (BFS/DFS/Union-Find) |
| «количество способов / минимум стоимости» | DP |
| «валидные скобки / monotonic / вложенность» | Stack |
| «префикс / автодополнение» | Trie |
| «пересечения / merge / расписание» | Intervals |

Узнавание паттерна — это 70% успеха. Дальше остаётся аккуратно закодить шаблон.

## Слой 5. Метод решения задачи

Это **процесс**, а не талант. Он одинаковый для всех задач:

**1. Понять условие.** Перефразируй своими словами. Выясни ограничения (размер `n`, диапазон значений, гарантии входа). Задай себе: что такое «валидный ответ», может ли он не существовать, могут ли быть дубликаты.

**2. Подобрать маленький пример.** Не тот, что в условии — свой. 3-5 элементов. Реши его руками. Именно руками — на бумажке или в голове. Часто после этого алгоритм становится очевиден.

**3. Найти паттерн.** Размер входа + ключевые слова + структура данных → паттерн из слоя 4. Если ничего не приходит — начни с брутфорса (перебрать все варианты), потом думай как его ускорить.

**4. Написать псевдокод.** Не сразу код на языке — сначала логика на русском или в виде пунктов. Это ловит баги на стадии замысла.

**5. Проверить на краях.** Пустой вход. Один элемент. Все одинаковые. Максимальный размер. Отрицательные. Переполнение `int`.

**6. Только теперь — код.**

**7. Прогнать свой пример в голове**, отслеживая переменные. Если ответ сходится — отправляй.

Большинство людей пропускают шаги 2 и 4 — и потом час отлаживают то, что можно было увидеть на бумажке за минуту.
